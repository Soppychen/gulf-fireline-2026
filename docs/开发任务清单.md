# 《海湾火线 2026》Demo 任务清单

版本：0.2  
Demo 范围：美国空军视角，第一关“闪电攻击”  
目标：完成一个可玩的 Web 纵向卷轴空战射击 Demo，并为后续 Android 和多阵营扩展打基础。

## 0. 当前实现记录

已完成一个可运行的 Web Demo 原型：

- 使用 TypeScript + Phaser 3 + Vite 搭建工程。
- 完成 BootScene、MenuScene、GameScene、ResultScene。
- 完成玩家、敌人、子弹、导弹、拾取物、Boss 车队实体。
- 完成键盘和触摸输入、暂停、主动护盾技能。
- 完成第一关时间轴刷怪、Boss 入场和胜负结算。
- 接入 `assets/` 内原创临时美术、背景、特效、音效和音乐资源。
- `npm run build` 已通过，浏览器本地冒烟测试无控制台 error/warn。

后续重点：

- 替换为正式可商用授权美术和音频。
- 增加自动化测试，覆盖评分、刷怪和碰撞等核心逻辑。
- 继续调优手机端性能、弹幕密度和 Boss 战节奏。

## 1. 里程碑

### M0：项目准备

- 确定技术栈：TypeScript + Phaser 3 + Vite。
- 建立项目目录和资源目录。
- 确定竖屏基准分辨率，例如 720x1280。
- 建立可运行的空白场景。
- 建立基础资源命名规范。

验收：

- 浏览器可打开空白游戏页面。
- 有加载场景、主菜单场景和游戏场景骨架。

### M1：核心手感

- 实现玩家飞机移动。
- 实现键盘输入：方向键/WASD、Shift 低速、Esc 暂停。
- 实现触摸拖拽移动。
- 实现自动射击。
- 实现子弹对象池。
- 实现敌人受击和销毁。

验收：

- 玩家能顺畅移动。
- 自动射击稳定。
- 同屏 100 个子弹不卡顿。
- 手机浏览器触摸操作不漂移。

### M2：第一关基础战斗

- 实现 3 种空中敌人：巡逻无人机、拦截无人机、轻型战机。
- 实现 3 种地面敌人：防空炮、导弹车、固定式 SAM 发射架。
- 实现敌方子弹和地对空追踪导弹。
- 实现玩家受击、无敌时间、生命系统。
- 实现简单关卡刷怪脚本。

验收：

- 第一关前 2 分钟可完整游玩。
- 玩家可受击、死亡、重开。
- 导弹有明显预警和可躲避窗口。

### M3：道具、评分和 Boss

- 实现火力升级、维修包、护盾。
- 实现导弹补给道具，拾取后让玩家获得有限弹药的机载追踪导弹。
- 实现连击和分数系统。
- 实现小 Boss“移动指挥车队”。
- 实现 Boss 血条和阶段切换。
- 实现胜利和失败结算。

验收：

- 第一关可从开始玩到结算。
- 评分显示击毁率、连击、受击次数、剩余生命、用时。
- Boss 至少有 2 个攻击阶段。

### M4：表现与可发布 Demo

- 接入正式或临时美术资源。
- 接入基础音乐和音效。
- 完成主菜单、暂停、结算 UI。
- 完成性能优化和移动端适配。
- 完成 README 和运行说明。

验收：

- Web Demo 可交给他人试玩。
- 桌面 60 FPS，手机 45 FPS 以上。
- 操作、音效、UI、关卡流程完整。

## 2. 程序任务

### 2.1 项目架构

- 创建 Phaser 3 + Vite + TypeScript 项目。
- 搭建场景系统：BootScene、MenuScene、GameScene、ResultScene。
- 搭建实体基类：Player、Enemy、Bullet、Missile、Pickup、Boss。
- 搭建数据驱动配置：aircraft、enemies、weapons、levels。
- 建立全局事件系统，用于分数、受击、拾取、暂停、胜利、失败。

注意事项：

- 不要把关卡刷怪逻辑硬编码在 GameScene 中，应放入 LevelConfig 或 SpawnSystem。
- 所有动态对象优先使用对象池。
- 输入系统要同时支持键盘和触摸，避免后期移动端返工。
- 数值配置应放在 data 文件，便于策划调整。

### 2.2 玩家系统

- 实现玩家移动边界。
- 实现自动射击计时。
- 实现武器等级。
- 实现玩家机载导弹弹药、冷却和自动发射。
- 实现生命、护盾、受击无敌。
- 实现主动技能：建议 Demo 先做 EMP 清弹或短时护盾。

注意事项：

- 玩家飞机视觉尺寸和碰撞判定要分离。
- 受击反馈要明显：闪烁、音效、短震动或屏幕轻微抖动。
- 自动射击音效要限制频率，避免声音过密。

### 2.3 敌人与弹幕

- 实现基础移动模式：直线、横向、俯冲、跟踪。
- 实现敌人射击模式：直线弹、扇形弹、追踪导弹。
- 实现地对空导弹流程：锁定预警、发射、追踪、失效。
- 实现导弹锁定优先级：敌方导弹追踪玩家，玩家导弹优先锁定 Boss 部件、导弹车、SAM 和轻型敌机。
- 实现地面单位固定在滚动背景上的表现。
- 实现敌人死亡掉落。

注意事项：

- 子弹速度和颜色必须可读。
- 追踪导弹要有转向上限，不能不可躲。
- 地对空导弹生成位置不能贴近玩家，发射前必须有 0.6-1.0 秒预警。
- 玩家导弹和敌方导弹必须用不同尾焰颜色、弹体轮廓和音效层级区分。
- 地面单位和背景要有层级区分。

### 2.4 关卡系统

- 实现纵向背景滚动。
- 实现时间轴刷怪。
- 实现第一关完整流程。
- 实现 Boss 入场、阶段、死亡和结算。

注意事项：

- 第一关前 30 秒要低压，给玩家学习窗口。
- 每种新威胁第一次出现时数量要少。
- Boss 战前应降低杂兵压力，避免信息过载。

### 2.5 UI 与存档

- 主菜单：开始游戏、设置、最高分。
- 游戏内 HUD：生命、分数、连击、技能冷却、Boss 血条。
- 暂停菜单：继续、重开、返回主菜单。
- 结算界面：评分、数据、重玩。
- localStorage 保存最高分和设置。

注意事项：

- 移动端按钮不要贴近屏幕边缘。
- HUD 不遮挡玩家常用移动区域。
- 文案避免真实政治宣传口吻。

## 3. 美术任务

### 3.1 飞机与单位

Demo 必需：

- 玩家飞机 AFX-35，含普通、受击闪烁、护盾状态。
- 巡逻无人机。
- 拦截无人机。
- 轻型敌机。
- 防空炮。
- 导弹车。
- 固定式 SAM 发射架。
- 雷达站。
- 移动指挥车队 Boss：通信车、防空车、护卫车。
- 玩家机载导弹。
- 敌方地对空导弹。

注意事项：

- 所有装备必须是原创变体，不直接描摹真实照片。
- 不使用真实国旗、军徽、部队编号。
- 玩家、敌人、中立背景的色彩需要一眼区分。
- 小尺寸下轮廓必须清楚，优先可读性。

### 3.2 场景与背景

Demo 第一关背景：

- 夜间沙漠。
- 山地峡谷。
- 公路和车队路线。
- 雷达站区域。
- 地下设施外围入口。

注意事项：

- 背景不要过亮或过花，避免干扰弹幕。
- 民用建筑只做背景，不作为攻击目标。
- 宗教建筑可以作为远景文化轮廓，但不作为可摧毁对象。
- 背景切片要能纵向循环或分段拼接。

### 3.3 特效

必需特效：

- 玩家机炮弹。
- 敌方普通弹。
- 导弹尾焰。
- 玩家蓝白导弹尾焰。
- 敌方红橙导弹尾焰。
- 导弹锁定预警框或屏幕边缘方向提示。
- 小爆炸、中爆炸、大爆炸。
- 拾取物光效。
- 护盾效果。
- EMP 清弹效果。
- Boss 阶段切换闪光。

注意事项：

- 玩家子弹、敌方子弹、拾取物颜色要明确分层。
- 爆炸不要长时间遮挡敌弹。
- 特效尺寸要控制，避免移动端性能问题。

### 3.4 UI 美术

必需 UI：

- 标题 Logo。
- 主菜单按钮。
- HUD 图标：生命、护盾、技能、分数。
- 暂停界面。
- 结算评分 C/B/A/S。
- Boss 血条。

注意事项：

- UI 风格现代军用界面即可，不要做过度复杂的拟真仪表。
- 字体必须清晰，移动端可读。
- 按钮状态要有普通、按下、禁用三种。

## 4. 音效与音乐任务

### 4.1 音乐

必需音乐：

- 主菜单循环音乐，30-60 秒。
- 第一关战斗循环音乐，90-120 秒。
- Boss 战循环音乐，60-90 秒。
- 胜利短音乐。
- 失败短音乐。

注意事项：

- 音乐要适合长时间循环，结尾需能自然回到开头。
- 第一关音乐偏快速电子/军鼓，不做沉重纪实战争片风格。
- Boss 音乐增加紧张感，但不要盖过导弹警告音。

### 4.2 音效

必需音效：

- 玩家机炮。
- 敌方机炮。
- 玩家机载导弹发射。
- 敌方地对空导弹发射。
- 敌方导弹锁定。
- 道具拾取。
- 玩家受击。
- 护盾吸收。
- EMP 释放。
- 小、中、大爆炸。
- Boss 入场。
- Boss 阶段变化。
- UI 点击。

注意事项：

- 自动射击音效要短、轻、低疲劳。
- 导弹锁定音要突出，但需设置冷却，避免连续尖叫。
- 多个爆炸同时触发时要做音量上限或随机挑选播放。
- Web 端注意浏览器自动播放限制，音乐应在玩家点击开始后播放。

## 5. 策划任务

### 5.1 第一关配置

- 写出第一关 0-240 秒刷怪时间轴。
- 确定每种敌人的血量、速度、伤害、得分。
- 确定道具掉落概率。
- 确定导弹补给的首次出现时间、掉落数量、弹药数量和发射冷却。
- 确定敌方 SAM 的预警时长、追踪时长、最大转向角和发射间隔。
- 确定 Boss 阶段血量和攻击节奏。
- 确定评分公式。

注意事项：

- 第一关不能一开始就高难。
- 每 20-30 秒引入一个变化点。
- 导弹和无人机不要同时高密度出现，除非进入 Boss 前高潮。
- 玩家第一次获得导弹补给后，应立刻安排一个高威胁地面目标让玩家感受到导弹价值。
- 玩家死亡原因必须可理解。

### 5.2 多阵营扩展设计

- 设计阵营数据结构：阵营名、默认机体、UI 色、任务类型、敌我单位表。
- 美国空军：突防、精确打击、隐身机。
- 以色列方面：高机动、短时间爆发、电子战。
- 伊朗方面：防空、无人机蜂群、阵地防御、反击关卡。

注意事项：

- 多阵营不是简单换皮，应有不同核心玩法。
- 每个阵营都应避免现实仇恨表达。
- 同一关卡可以从不同阵营视角复用，但胜利目标和敌人配置不同。

### 5.3 文案

- 主菜单文案。
- 任务简报。
- 关卡内提示。
- 结算文案。
- 设置项文案。

注意事项：

- 不使用现实人物姓名作为目标。
- 不使用“消灭某民族/宗教/国家”等表达。
- 第一关“斩首任务”在内部可作为设计代号，对外文案建议称为“高价值指挥节点突袭”。

## 8. 风险清单

### 8.1 题材风险

风险：现实冲突题材容易被理解为宣传或仇恨表达。  
处理：架空化阵营标识、目标、地图和任务文本；多阵营视角保持克制。

### 8.2 美术版权风险

风险：直接使用新闻照片、军工厂商图片或真实徽章可能侵权。  
处理：所有资产原创绘制，现实装备仅作为轮廓灵感。

### 8.3 玩法复杂度风险

风险：现代武器类型过多导致 Demo 做不完。  
处理：第一版只做机炮、追踪导弹、护盾、EMP、少量敌人。

### 8.4 移动端性能风险

风险：弹幕和爆炸过多导致手机卡顿。  
处理：对象池、限制同屏数量、使用 spritesheet、减少透明大图叠加。

### 8.5 操作适配风险

风险：桌面好玩，手机难操作。  
处理：从 M1 起实现触摸拖拽，不等后期移植。

## 9. 建议分工

最小团队：

- 程序 1 人：核心玩法、关卡、UI、性能。
- 美术 1 人：飞机、敌人、背景、特效、UI。
- 音效/音乐 1 人，可兼职或外包。
- 策划/制作 1 人：数值、关卡节奏、验收、内容把关。

单人开发优先级：

1. 先用占位图完成完整玩法。
2. 再做第一关完整时间轴。
3. 再替换玩家、敌人、子弹和背景资源。
4. 最后补音乐、音效、菜单和结算。

## 10. 下一步建议

下一步可以先建立 Web 项目骨架，然后用简单几何图形完成第一关原型。只要移动、自动射击、敌人生成和碰撞手感成立，再逐步替换美术和音效。
